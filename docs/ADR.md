# Architecture Decision Records (ADR)

## ADR-001: RAG 및 Vector DB의 전면 폐기와 LLM Wiki 패턴 도입
- **결정**: 정보를 매번 재조립하지 않고, 소스가 추가될 때마다 의미를 추출하여 **'지속적으로 교차 참조가 축적되는 마크다운 위키(LLM Wiki)'** 형태로 영구 보존함. Vector DB(RAG)를 프로젝트에서 완전히 배제하고, Neo4j 기반 에이전틱 순회(Agentic Traversal) 방식을 채택하여 지식 검색 아키텍처를 전면 개편함.
- **이유**: 기존 RAG 시스템은 질문이 들어올 때마다 파편화된 문서 조각들을 불러와서 처음부터 다시 조립(Rediscover from scratch)하므로 복잡한 문맥 파악에 한계가 있고 매번 LLM 비용이 발생함. 수학적 유사도 기반의 한계를 피해야 함.
- **트레이드오프**: pgvector 기반의 간편한 유사도 검색을 포기함. 위키 컴파일 과정에서 LLM 호출 비용이 선행적으로 발생하지만, 이후 조회 시에는 에이전트가 잘 정리된 위키의 '표지판'을 따라 순회하기만 하면 되므로 진정한 의미의 지식 추론이 가능해짐.

## ADR-002: Ingest 시 Index & Log 파일 관리와 증분 업데이트
- **결정**: LLM이 Ingest 작업을 수행할 때마다 `index.md`(콘텐츠 중심 카탈로그)와 `log.md`(연대기적 작업 기록)를 업데이트하도록 스키마에 정의. 변경된 델타(Delta)와 인접 노드만 탐색하여 컴파일하는 증분 처리 파이프라인 구축.
- **이유**: 수백 개의 위키 파일을 매번 전체 탐색하는 것은 매우 비효율적임.
- **트레이드오프**: index.md/log.md에 대한 동시 수정 시 레이스 컨디션 발생 가능. Redis 분산 락(RedissonClient)으로 해결하나, 락 경합 시 약간의 처리 지연이 생길 수 있음.

## ADR-003: 데이터 정합성을 위한 트랜잭셔널 아웃박스 패턴 적용
- **결정**: 사용자나 에이전트의 노트 추가 이벤트를 관계형 DB(MSSQL 아웃박스 테이블)에 우선 기록한 뒤, Kafka 메시징을 통해 Worker들이 마크다운 갱신과 DB 쿼리를 병렬로 반영토록 하여 최종 일관성 확보.
- **이유**: 로컬 마크다운 위키(파일 시스템)와 그래프 DB(Neo4j) 동시 업데이트 시 부분 장애(Partial Failure) 발생 위험이 있음.
- **트레이드오프**: 2PC(2-Phase Commit) 대비 구현 복잡도는 낮지만, 최종 일관성(Eventual Consistency) 모델이므로 수 초간 데이터 불일치가 존재할 수 있음. 강한 일관성을 포기.

## ADR-004: Query 분석 결과의 재귀적 영속화 (Filed Back into Wiki)
- **결정**: 사용자의 탐색(Query) 오퍼레이션에서 생성된 가치 있는 분석, 비교, 통찰을 단순 응답에 그치지 않고 새로운 위키 페이지로 생성하여 지식 베이스에 다시 주입(Filed back)하는 재귀적 지식 증식 구조 채택.
- **이유**: 사용자의 심층적인 질문(Query)을 통해 도출된 통찰이나 훌륭한 분석 결과가 휘발성 채팅 이력으로 사라지는 것을 방지함.
- **트레이드오프**: 자동 File-back으로 인해 위키에 저품질 페이지가 누적될 위험이 있음. Lint 데몬의 STALE_DATA 감지로 완화하나, 초기에는 인간의 큐레이션이 필요함.

## ADR-005: 비용 절감형 자가 치유 지식 린터 (Self-Healing Knowledge Linter) 데몬
- **결정**: Neo4j의 그래프 구조를 활용해 '최신 변경이 발생한 노드의 의존성 하위 트리'만 추출. 데몬이 정기적으로 Lint 오퍼레이션을 수행하여 고아 문서나 모순된 내용을 찾아내고 스스로 건강(Health-check)을 유지하도록 자동화.
- **이유**: 위키가 커질수록 모순된 주장, 구식 데이터, 고아 문서(Orphan pages), 누락된 교차 참조 등 유지보수 빚이 쌓임. 전체를 스캔하면 O(N)의 비용 폭탄 발생.
- **트레이드오프**: 서브그래프만 탐색하므로 변경의 영향이 먼 노드에 파급된 경우 감지하지 못할 수 있음. 주기적인 FULL 스캔(주 1회)으로 보완.

## ADR-006: ORM 대신 MyBatis 선택
- **결정**: JPA/Hibernate 대신 MyBatis를 SQL 매퍼로 채택.
- **이유**: 아웃박스 패턴의 복잡한 상태 관리 쿼리와 배치 처리에서 SQL을 직접 제어할 필요가 있음. MSSQL 고유 기능(MERGE, OUTPUT 절 등) 활용에 유리.
- **트레이드오프**: Entity 자동 매핑의 편의성을 포기. XML 매퍼 파일 관리 부담이 생기지만, 쿼리 성능 튜닝과 디버깅이 훨씬 용이함.

## ADR-007: 다변화된 온톨로지(Edge Types) 동적 주입
- **결정**: 에이전트가 개념 간의 엣지를 생성할 때 단일 관계(`RELATES_TO`)로 퉁치지 않고, `EXTENDS`, `CONTRADICTS`, `DEPENDS_ON`, `EXPLAINS` 등 구체적인 논리적 관계(Type)를 동적으로 판단하여 주입함.
- **이유**: 단순 연결만으로는 에이전트가 탐색 중 문맥(이게 반박인지 확장인지)을 이해할 수 없음. 입체적이고 의미 있는 온톨로지 지식망을 형성하기 위함.
- **트레이드오프**: LLM이 관계의 종류를 정확히 판별하기 위한 프롬프트 복잡도가 상승하며, Neo4j 조회 쿼리 시 동적 매핑 로직이 필요함.

## ADR-008: 볼트별 API 키 관리(BYOK) 폐기 및 전역 API 키 통합
- **결정**: 개별 워크스페이스/볼트마다 암호화하여 DB에 보관하던 사용자 API 키(BYOK) 구조를 폐기하고, 시스템 환경 변수(`OPENAI_API_KEY`)를 통해 단일 전역 API 키를 사용하여 LLM 서비스를 구동함 (이후 ADR-014의 DB 자격증명 관리 기능으로 확장 발전됨).
- **이유**: 각 볼트별 API 키 저장 관리의 오버헤드가 크고 보안 취약 노출 지점이 많으며, 다중 볼트 이동 시 통신 장애나 키 누락 이슈를 방지하기 위함.
- **트레이드오프**: 멀티테넌트 BYOK 유연성은 줄어들었으나, 호스팅 서버 단위의 단일 계정 및 로컬 금고(Vault) 아키텍처에 적합한 강력하고 안정적인 실행 토대를 확보함.

## ADR-009: 위키 페이지 조회 시 Neo4j 연동 한글/공백 타이틀 및 Slug 동적 해소
- **결정**: 위키 페이지 조회 엔드포인트(`GET /api/v1/wiki/{conceptName}`) 호출 시, `conceptName`이 파일명(영문 Slug)이 아닌 한글 명칭이나 띄어쓰기가 있는 명칭(예: `하네스 프레임워크`)이어도 Neo4j 그래프 데이터베이스를 실시간 쿼리하여 매핑된 영문 Slug(`harness-framework`)를 찾아내고, 해당 영문 Slug 기반의 마크다운 파일을 로드하도록 함.
- **이유**: 에이전트가 마크다운 본문에 Zettelkasten 형식의 한글 교차 참조(`[[하네스 프레임워크]]`)를 생성할 때, 파일명은 영문 Slug로 생성되어 있어서 한글로만 된 링크 클릭 시 404 이슈가 발생함. 이를 방지하고 한글 교차 참조의 탐색 편의성을 극대화하기 위해 실시간 해소 레이어를 구축함.
- **트레이드오프**: 마크다운 파일 직접 조회 시 매번 그래프 DB 조회가 선행될 수 있으나, 영문 Slug 형식인 경우에는 DB 쿼리를 타지 않도록 우선 파일 검증(Direct lookup)을 거치도록 최적화하여 Neo4j 조회 오버헤드를 최소화함.

## ADR-010: LLM Wiki 패턴을 따르는 파일 기반의 가벼운 컨텍스트 검색(File-based Context Retrieval) 시스템 도입
- **결정**: 질문(Query) 오퍼레이션 수행 시, 벡터 데이터베이스나 임베딩 없이 로컬 지식 위키 디렉토리(`wiki/concepts`, `entities`, `insights`) 내의 모든 마크다운 파일들의 목록을 읽고, gpt-4o-mini 호출을 사용하여 질문에 가장 관련성 높은 3~5개의 위키 문서를 가볍게 선정한 후 그 내용을 최종 LLM 프롬프트에 컨텍스트로 주입함.
- **이유**: RAG의 전면 폐기로 인해 사용자가 질문을 던졌을 때 일반적인 상식을 기반으로 답변하는 현상을 방지해야 함. 무겁고 복잡한 벡터 DB 구축 대신, 텍스트 기반 Zettelkasten 마크다운 목록을 OpenAI로 1차 필터링하여 컨텍스트를 주입함으로써 "LLM Wiki (Second Brain)" 사상을 컴팩트하게 만족시킴.
- **트레이드오프**: 1차 파일 선정을 위한 LLM 비용이 추가로 발생하나, 질문의 성격과 관련 파일을 매핑하는 것이 gpt-4o-mini의 가볍고 저렴한 호출로 처리 가능하므로 아키텍처의 단순함을 유지하면서도 높은 정확도를 제공함.

## ADR-011: LLM 환각 방지를 위한 위키 페이지 생성 시 relatedConcepts 교차 검증 적용
- **결정**: LlmService의 위키 생성(`generateWikiPages`) 로직 수행 시, LLM이 반환하는 JSON 스키마 구조 내의 `relatedConcepts`를 그대로 위키 파일에 기입하지 않고, 입력된 그래프 데이터의 실제 `links` 관계 정보(온톨로지 에지)와 교차 검증(Cross-reference)하여 실제 관계가 존재하는 노드만 수집 및 바인딩하도록 강제함.
- **이유**: LLM은 위키 페이지의 "관련 개념들"을 기재할 때, 실제 온톨로지 상에 존재하지 않거나 관계가 형성되지 않은 개념들까지 임의로 상상해서 기입하는 환각(Hallucination) 현상을 보임. 이는 그래프 DB와 마크다운 파일 간의 정합성을 깨뜨리고, 데이터 무결성 오류를 초래함.
- **트레이드오프**: LLM이 제안한 연관 개념 중 온톨로지 상에 아직 등록되지 않은 유용한 개념이 제외될 수 있으나, 지식의 신뢰성 및 그래프-위키 데이터 간 100% 무결성을 유지하는 것이 더 높은 비즈니스 가치를 지님.

## ADR-012: 워크스페이스 데이터 삭제 처리 범위 확대 (물리 디렉토리 전체 삭제)
- **결정**: 워크스페이스 데이터 삭제 시, 기존에 Neo4j 데이터 삭제와 워크스페이스 내부의 `wiki` 폴더만 삭제하던 방식에서 워크스페이스의 물리적 루트 디렉토리 자체(`workspaces/{workspace_id}`)를 하위 파일 및 폴더 전체와 함께 재귀적으로 완전 제거하도록 정책을 강화함.
- **이유**: 단순 `wiki` 폴더 삭제만으로는 워크스페이스의 최상위 빈 폴더 및 기타 메타데이터 파일들이 찌꺼기로 남아, 추후 동일 이름으로 테넌트 재생성 시 정합성 에러를 일으키거나 보안상 흔적이 남는 등의 문제가 발생함.
- **트레이드오프**: 실수로 삭제 시 데이터 복구가 절대 불가능하므로, API 인터페이스 수준 및 컨트롤러 보안 레이어에서 철저한 JWT 테넌트 정합성 가드를 적용해야 함.

## ADR-013: LangChain4j를 활용한 Agentic Graph Traversal 도입
- **결정**: LLM이 1차 필터링된 위키 문서를 단발성으로 읽는 대신, LangChain4j의 `AiServices`와 `@Tool`을 활용하여 LLM이 자율적으로 그래프를 탐색(`searchGraph`), 이웃 노드 조회(`getNodeContext`), 마크다운 페이지 조회(`readWikiPage`), 외부 Notion 페이지 조회(`readNotionPage`)를 호출하고 질문에 대답하도록 구조 개선.
- **이유**: 단발성 LLM 호출로는 그래프 데이터베이스가 가진 복잡한 관계 정보와 연결된 위키 문서들의 깊은 컨텍스트를 완벽히 이해하기 어렵기 때문.
- **트레이드오프**: 에이전트의 멀티턴 툴 호출(Multi-turn Tool Calling)로 인해 LLM 비용이 다소 증가하고 응답 속도가 약간 늦어질 수 있으나, 답변의 깊이와 정합성은 획기적으로 상승함.

## ADR-014: API 토큰 DB 암호화 마이그레이션 (BYOK)
- **결정**: 사용자가 입력하는 Notion, GitHub 등의 연동용 API 토큰(Credentials)을 프론트엔드의 localStorage에 저장하는 대신 백엔드 MSSQL 데이터베이스(`WorkspaceCredentials` 테이블)에 저장하되, **AES-256 암호화**를 적용해 기밀성을 유지하도록 마이그레이션함.
- **이유**: 프론트엔드 브라우저 스토리지에 API 키가 평문으로 남을 경우 XSS 및 토큰 유출 공격에 매우 취약함. 백엔드에서 격리 및 암호화하여 관리함으로써 보안 신뢰 경계를 확보함.
- **트레이드오프**: 백엔드 DB 암호화/복호화 오버헤드가 발생하지만, 서버 메모리 캐싱 및 서비스 최적화를 통해 리드 부하를 최소화함.

## ADR-015: Redis JWT Blacklisting 및 회원 영구 탈퇴 처리 정책
- **결정**: B2B 요구사항에 따라 세션 제어 및 안전한 계정 삭제를 처리하기 위해,
  1. 로그아웃 시 JWT 토큰의 잔여 수명 동안 Redis에 블랙리스트 키(`blacklist:<token>`)를 적재하고 `JwtInterceptor`에서 가로채어 401 Unauthorized 에러를 반환.
  2. 회원 영구 탈퇴 시 해당 계정의 Neo4j 노드, 물리적 마크다운 위키 디렉토리, DB 사용 기록을 트랜잭션 단위로 일괄 영구 소멸시키는 엄격한 데이터 소거 정책 수립.
- **이유**: B2B 프라이버시 및 GDPR/B2B 보안 규정을 충족하고, 로그아웃한 토큰의 재사용을 원천 차단하기 위함.
- **트레이드오프**: 회원 탈퇴 시 한 번 지워진 데이터는 복구가 불가능하므로, 프론트엔드 확인 모달을 통한 경고 및 안전 가이드라인이 필요함.

## ADR-016: AI Gaze Tracking을 위한 실시간 SSE 기반 Node Highlight 시각화
- **결정**: AI 에이전트가 `getNodeContext` 및 `readWikiPage` 같은 Graph Traversal 도구를 수행하여 지식을 탐색할 때, 해당 노드 ID를 실시간 SSE 알림 스트림(`/v1/notifications/sse` 내 `ai_reading` 이벤트)을 통해 프론트엔드로 브로드캐스트하고, 프론트엔드는 이를 수신해 2D 그래프 상에서 Emerald 500 테두리와 에메랄드 글로우, 초록색 텍스트로 3초 동안 시각적 하이라이트를 제공함.
- **이유**: 백그라운드나 긴 대기 시간이 발생하는 AI 그래프 탐색 과정에서 AI가 어떤 지식을 읽고 추론하고 있는지를 사용자에게 실시간 시각적 피드백으로 제공하여 AI의 탐색 방식에 투명성과 생동감을 부여함.
- **트레이드오프**: 도구 호출이 빈번할 경우 잦은 SSE 이벤트 발행 및 렌더링 부하가 발생할 수 있으므로, 3초 간의 timeout 처리를 도입해 클라이언트 렌더링 오버헤드를 완화함.

## ADR-017: B2B 멀티테넌시 워크스페이스 물리적 및 논리적 격리 정책
- **결정**: 여러 사용자가 동시에 사용하는 B2B 환경에서 타인 워크스페이스에 대한 접근을 차단하기 위해, 각 워크스페이스의 물리적 경로 및 Neo4j의 `workspaceId` 식별자에 사용자의 username을 조합한 `ws-username` 및 `username_workspace-name` 접두사 구조를 강제 부여하여 조회 및 생성을 격리함.
- **이유**: 타인의 워크스페이스 명이 유출되거나 권한이 없는 워크스페이스에 접근하는 공격을 아키텍처 수준에서 격리 및 예방하기 위함.
- **트레이드오프**: 워크스페이스 명칭이 다소 길어지고 조회 조건 검증 필터 오버헤드가 발생하지만 데이터 격리의 안정성을 위해 감수함.
